# Задание 4: AI система за управление на инциденти

## Седмица 2 - ООП и събития

### 1. Цел на втория етап

Вторият етап реализира първата работеща ООП структура на проекта. Фокусът е върху класовете, комуникацията между обектите, събитията и callback функциите. Базата данни и файловата обработка ще се разширят в следващия етап.

### 2. Реализирани класове

| Клас | Файл | Роля |
| --- | --- | --- |
| `Incident` | `src/models/incident.py` | Модел на инцидент: заглавие, описание, категория, засегнати потребители, приоритет и статус. |
| `Technician` | `src/models/technician.py` | Модел на техник, който може да бъде назначен към инцидент. |
| `IncidentEventArgs` | `src/models/event_args.py` | Custom event args клас с данни за възникнало събитие. |
| `EventManager` | `src/events/event_manager.py` | Event dispatcher, който регистрира слушатели и изпраща събития. |
| `IncidentClassifier` | `src/ai/incident_classifier.py` | Първична rule-based AI логика за предсказване на приоритет. |
| `IncidentManager` | `src/services/incident_manager.py` | Контролер, който създава инциденти, извиква AI класификатора и изпраща събития. |
| `AppLogger` | `src/services/app_logger.py` | Слушател, който записва и извежда логове при събития. |
| `NotificationService` | `src/services/notification_service.py` | Слушател, който симулира уведомяване на екипа при критичен инцидент. |

### 3. Събития

| Събитие | Кога се извиква | Слушатели |
| --- | --- | --- |
| `on_incident_created` | След създаване на нов инцидент. | `AppLogger.log_event` |
| `on_priority_changed` | Когато AI класификаторът промени първоначалния приоритет. | `AppLogger.log_event` |
| `on_critical_detected` | Когато предсказаният приоритет е `high`. | `AppLogger.log_event`, `AppLogger.log_critical_incident`, `NotificationService.notify_support_team` |

Изискването за минимум два слушателя на едно събитие е изпълнено чрез `on_critical_detected`, което има три слушателя.

### 4. Комуникация между обектите

1. `main.py` създава `EventManager`, `IncidentClassifier`, `AppLogger`, `NotificationService` и `IncidentManager`.
2. `AppLogger` и `NotificationService` се регистрират като callback слушатели чрез `EventManager.subscribe`.
3. `IncidentManager.create_incident` създава нов обект `Incident`.
4. `IncidentManager` подава инцидента към `IncidentClassifier`.
5. `IncidentClassifier` връща предсказан приоритет: `low`, `medium` или `high`.
6. `IncidentManager` записва инцидента в списък и изпраща подходящите събития чрез `EventManager.notify`.
7. `EventManager` извиква всички callback функции, регистрирани за съответното събитие.

### 5. Callback функции

Callback функциите са методи, които се подават към `EventManager` и се извикват при събитие:

```python
event_manager.subscribe("on_incident_created", logger.log_event)
event_manager.subscribe("on_critical_detected", logger.log_event)
event_manager.subscribe("on_critical_detected", logger.log_critical_incident)
event_manager.subscribe("on_critical_detected", notifications.notify_support_team)
```

Така проектът използва Observer pattern в Python стил.

### 6. Първична AI логика

`IncidentClassifier` използва точки:

- много засегнати потребители;
- рискова категория: `security`, `database`, `network`;
- сериозни ключови думи: `outage`, `breach`, `data loss`, `down`, `malware`;
- средно рискови ключови думи: `slow`, `error`, `failed`, `timeout`, `locked`.

Резултат:

| Точки | Приоритет |
| --- | --- |
| `0-2` | `low` |
| `3-5` | `medium` |
| `6+` | `high` |

### 7. Демонстрация

Проектът може да се стартира с:

```bash
python -m src.main
```

Очаквано поведение:

- създава се критичен инцидент `Database outage`;
- класификаторът го определя като `high`;
- изпращат се събитията `on_incident_created`, `on_priority_changed` и `on_critical_detected`;
- `on_critical_detected` се обработва от повече от два слушателя;
- създава се втори примерен инцидент с нисък приоритет.
